📝 1. Introducción a las Herramientas del Análisis Estructurado

El análisis estructurado es una metodología tradicional que considera los datos y los procesos que los transforman como entidades separadas. Esta aproximación utiliza un conjunto específico de herramientas que permiten modelar de manera clara y precisa los requerimientos de un sistema, proporcionando una base sólida para el desarrollo de software confiable y mantenible.

1.1 Evolución Histórica del Análisis Estructurado

El análisis estructurado surgió en la década de 1970 como respuesta a la crisis del software y la necesidad de metodologías más rigurosas para el desarrollo de sistemas. Los pioneros como Tom DeMarco, Ed Yourdon y Chris Gane establecieron los fundamentos teóricos que aún utilizamos hoy.

1.2 Principios Fundamentales

Principio de Separación: Los datos y los procesos se modelan independientemente para reducir acoplamiento
Principio de Jerarquía: Los sistemas complejos se descomponen en niveles manejables de abstracción
Principio de Balanceo: Cada nivel de descomposición debe ser consistente con el nivel superior
Principio de Trazabilidad: Todos los elementos deben ser rastreables a través de los artefactos
Integración Metodológica: Estas herramientas trabajan de manera coordinada y deben mantener consistencia entre sí. La trazabilidad entre artefactos es fundamental para garantizar la calidad del análisis. La metodología requiere un enfoque disciplinado pero flexible para adaptarse a las necesidades específicas de cada proyecto.

1.3 Aplicabilidad en el Contexto Actual

Aunque surgió en la era del desarrollo estructurado, el análisis estructurado mantiene relevancia en el contexto actual por varias razones fundamentales:

  • Sistemas Legacy: Muchos sistemas críticos fueron desarrollados usando análisis estructurado
  • Regulaciones y Auditorías: Industrias reguladas requieren documentación detallada
  • Sistemas Complejos: La descomposición jerárquica sigue siendo efectiva para sistemas grandes
  • Integración de Sistemas: Los DFD son excelentes para modelar interfaces entre sistemas
  • Formación Conceptual: Proporciona bases sólidas para entender otros paradigmas

💡 Síntesis Conceptual

Según Pressman, "el propósito de los diagramas de flujo de datos es proveer un puente semántico entre los usuarios y los desarrolladores de sistemas". Esta capacidad de comunicación es lo que hace tan valiosas estas herramientas en el proceso de análisis.

Objetivo de la clase: Dominar el uso integrado de las cuatro herramientas para crear modelos de análisis coherentes, trazables y mantenibles que sirvan como base sólida para el diseño e implementación de sistemas de información robustos.

🎯 2. Diagrama de Contexto

2.1 Definición y Propósito Fundamental

El diagrama de contexto, también conocido como DFD de Nivel 0, es la representación más alta del sistema en el análisis estructurado. Su función principal es definir claramente los límites del sistema y mostrar cómo interactúa con las entidades externas de su entorno. Esta herramienta constituye el punto de partida para todo el análisis posterior y sirve como contrato inicial entre analistas, desarrolladores y stakeholders.

Objetivo Principal: Establecer el alcance del sistema identificando qué está dentro y qué está fuera de los límites del sistema a desarrollar, creando una vista de "caja negra" que enfoca en las interfaces sin revelar la complejidad interna.

2.4 Reglas de Construcción Detalladas

Regla 1 - Unicidad del Proceso: Un diagrama de contexto debe contener exactamente un proceso central que represente todo el sistema
Regla 2 - Ausencia de Almacenes: No debe mostrar almacenes de datos internos, solo el flujo de información hacia y desde el exterior
Regla 3 - Conectividad Externa: Todos los flujos deben conectar el sistema central con entidades externas, nunca entre entidades externas
Regla 4 - Nomenclatura Descriptiva: Cada flujo debe tener un nombre que describa claramente qué información contiene
Regla 5 - Bidireccionalidad Explícita: Si hay comunicación bidireccional, debe representarse con flujos separados claramente etiquetados

🌊 3. Diagramas de Flujo de Datos (DFD)

3.1 Fundamentos Teóricos de los DFD

Los Diagramas de Flujo de Datos representan el corazón del análisis estructurado. Adoptan un punto de vista "entrada-proceso-salida" para modelar cómo los datos fluyen a través del sistema y cómo son transformados por los diferentes procesos. Esta aproximación funcional permite abstraer la complejidad organizacional y tecnológica para enfocarse en las transformaciones esenciales de información.

Definición Formal: Un DFD es una representación gráfica que muestra el flujo de datos a través de un sistema de información, los procesos que transforman estos datos, los lugares donde se almacenan, y las fuentes y destinos externos de información. Proporciona una vista lógica independiente de la implementación física.

3.4 Reglas de Construcción Detalladas

Regla de Balanceo Fundamental: Las entradas y salidas de un proceso en un nivel deben coincidir exactamente con las entradas y salidas de su descomposición en el nivel inferior
Regla de Numeración Jerárquica: Los procesos se numeran sistemáticamente (1, 2, 3 en nivel 1; 1.1, 1.2, 1.3 para subprocesos del proceso 1)
Regla de Etiquetado Descriptivo: Todos los flujos, procesos y almacenes deben tener nombres descriptivos y únicos que reflejen su propósito funcional
Regla de Conectividad: Cada elemento debe estar conectado al menos a otro elemento; no pueden existir elementos aislados
Regla de Simplicidad: Evitar cruces de líneas innecesarios y mantener el diagrama legible y comprensible

3.8 Técnicas de Descomposición

Criterios para la Descomposición de Procesos

Un proceso debe descomponerse cuando presenta una o más de las siguientes características:

  • Alta complejidad funcional: Realiza múltiples transformaciones no cohesivas
  • Múltiples responsabilidades: Maneja diferentes tipos de datos o funciones
  • Difícil especificación: La lógica interna es demasiado compleja para una sola especificación
  • Reutilización: Contiene subprocesos que podrían ser reutilizados
  • Mantenibilidad: La descomposición facilita modificaciones futuras

📚 4. Diccionario de Datos

Concepto y Propósito

El diccionario de datos es un listado organizado que contiene definiciones precisas y rigurosas de todos los datos pertinentes al sistema. Su objetivo es que el analista tenga un entendimiento completo de todos los datos que forman parte de los procesos del sistema.

Definición: El diccionario de datos describe el significado de los flujos y almacenes que se muestran en los DFD, especificando la composición de los paquetes de datos, sus valores y unidades relevantes.

Notación del Diccionario de Datos

El diccionario de datos utiliza una notación específica para definir la estructura y composición de los datos:

Símbolo Significado Descripción Ejemplo
= Está compuesto de Define la composición de un elemento de datos Nombre = Primer_nombre + Apellido
+ Y (concatenación) Une elementos que aparecen juntos Dirección = Calle + Número + Ciudad
( ) Opcional Elemento que puede estar presente o no Nombre = Primer_nombre + (Segundo_nombre) + Apellido
{ } Iteración Elemento que se repite cero o más veces Estudiantes = {Nombre + Legajo}

Ejemplos Prácticos de Definiciones

Ejemplo de Diccionario de Datos: Las definiciones en el diccionario de datos establecen la estructura precisa de la información que fluye por el sistema. Cada entrada describe elementos como comandos de usuario que pueden ser de configuración, activación o consulta, junto con sus parámetros específicos. Los registros de estado de sensores incluyen identificadores únicos, tipos de sensor (puerta, ventana, movimiento, humo), ubicación física y estados posibles (activado, desactivado, alarma, error). Esta documentación formal garantiza la consistencia en el diseño y facilita la implementación posterior del sistema, proporcionando especificaciones detalladas de formatos, rangos de valores y estructuras de datos complejas.

⚙️ 5. Especificaciones de Procesos

Definición y Propósito

Las especificaciones de procesos, también conocidas como minispec o PSPEC (Process Specification), describen la lógica de transformación de los procesos primitivos identificados en los DFD. Su propósito es definir exactamente qué hace cada proceso con los datos de entrada para producir los datos de salida.

Objetivo: Proporcionar una descripción precisa, no ambigua y verificable de la lógica interna de cada proceso primitivo del sistema.

Técnicas de Especificación

1. Lenguaje Estructurado (Pseudocódigo)

Especificación con Lenguaje Estructurado: El pseudocódigo proporciona una descripción algorítmica precisa de la lógica del proceso, utilizando estructuras de control estándar como condicionales y bucles. En este ejemplo, se especifica el comportamiento de validación de contraseñas, donde primero se verifica contra el password maestro y, en caso de no coincidir, se recorre una tabla de passwords secundarios. Esta técnica permite especificar procesos complejos de manera comprensible para analistas y programadores, manteniendo la independencia del lenguaje de implementación mientras garantiza que toda la lógica del negocio quede completamente documentada y sea verificable.

📝 Proceso 2.2: Calcular Capacidad del Equipo

Número de Proceso: 2.2 (subproceso de 2.0 Planificar Sprint)

Tipo: Proceso Primitivo (no se descompone más)

Entradas:
  • Validated Dates: Fechas de inicio y fin del sprint (desde proceso 2.1)
  • Team Capacity Data: Disponibilidad de cada miembro (desde almacén D6)
Salidas:
  • Available Capacity: Capacidad total del equipo en story points
Lógica del Proceso (Pseudocódigo Estructurado):
PROCESO Calcular_Capacidad_Equipo
ENTRADA:
    sprint_start_date: Date
    sprint_end_date: Date
    team_members: Array[TeamMember]
SALIDA:
    total_capacity: Integer (story points)

INICIO
    // 1. Calcular días hábiles del sprint
    working_days ← Contar_Dias_Habiles(sprint_start_date, sprint_end_date)

    // 2. Descontar feriados
    holidays ← Obtener_Feriados(sprint_start_date, sprint_end_date)
    working_days ← working_days - COUNT(holidays)

    // 3. Calcular capacidad individual de cada miembro
    total_capacity ← 0

    PARA CADA member EN team_members HACER
        // 3.1 Obtener disponibilidad del miembro
        member_availability ← member.availability_percentage / 100

        // 3.2 Descontar vacaciones y ausencias
        member_absences ← Contar_Ausencias(member, sprint_start_date, sprint_end_date)
        member_working_days ← working_days - member_absences

        // 3.3 Calcular horas disponibles
        daily_hours ← 8  * constante: 8 horas por día *
        focus_factor ← 0.7  * constante: 70% del tiempo en desarrollo puro *
        member_hours ← member_working_days * daily_hours * member_availability * focus_factor

        // 3.4 Convertir horas a story points según velocidad histórica
        SI member.historical_velocity > 0 ENTONCES
            hours_per_story_point ← member.historical_velocity
        SINO
            hours_per_story_point ← 6  * default: 6 horas por story point *
        FIN SI

        member_capacity ← REDONDEAR(member_hours / hours_per_story_point)

        // 3.5 Acumular capacidad
        total_capacity ← total_capacity + member_capacity

        // 3.6 Registrar capacidad individual para auditoría
        Registrar_Capacidad_Individual(member.id, member_capacity)
    FIN PARA

    // 4. Aplicar factor de contingencia
    contingency_factor ← 0.9  * reservar 10% para imprevistos *
    total_capacity ← REDONDEAR(total_capacity * contingency_factor)

    // 5. Validar capacidad mínima
    SI total_capacity < 10 ENTONCES
        Generar_Alerta("Capacidad del equipo muy baja para el sprint")
    FIN SI

    RETORNAR total_capacity
FIN PROCESO
Reglas de Negocio:
  • RN-01: El focus factor estándar es 70% (30% para reuniones, emails, interrupciones)
  • RN-02: La capacidad debe ajustarse por factor de contingencia de 90% (reserva 10% para imprevistos)
  • RN-03: Si un miembro no tiene velocidad histórica, usar 6 horas por story point como default
  • RN-04: Generar alerta si la capacidad total es < 10 story points
  • RN-05: Los días feriados se obtienen del calendario configurado por el Project Manager
Casos de Excepción:
  • E-01: Si working_days ≤ 0 → Lanzar error "Rango de fechas inválido"
  • E-02: Si team_members está vacío → Lanzar error "No hay miembros asignados al equipo"
  • E-03: Si member_absences > working_days → Usar capacidad = 0 para ese miembro

📝 Proceso 5.1: Generar Burndown Chart

Número de Proceso: 5.1 (subproceso de 5.0 Generar Reportes)

Tipo: Proceso Primitivo

Entradas:
  • Sprint Data: Información del sprint activo (desde D2)
  • Task Data: Estado actual de todas las tareas (desde D3)
Salidas:
  • Burndown Chart: Gráfico de progreso del sprint
Lógica del Proceso (Tabla de Decisión):
Condición Caso 1 Caso 2 Caso 3 Caso 4
Sprint Status Planning Active Active Closed
Progreso vs Ideal N/A On Track Behind N/A
Días Restantes > 10 > 2 ≤ 2 0
Acción: Generar Gráfico ✅ ✅ ✅ ✅
Acción: Alerta de Riesgo ❌ ❌ ✅ ❌
Acción: Proyección Final ❌ ✅ ✅ ❌
Acción: Archivar Métricas ❌ ❌ ❌ ✅
Algoritmo de Cálculo:
PROCESO Generar_Burndown_Chart
ENTRADA:
    sprint: Sprint
    tasks: Array[Task]
SALIDA:
    chart: BurndownChart

INICIO
    // 1. Calcular línea ideal
    total_story_points ← sprint.velocity_target
    sprint_days ← Calcular_Dias_Habiles(sprint.start_date, sprint.end_date)
    ideal_burn_rate ← total_story_points / sprint_days

    ideal_line ← []
    PARA day ← 0 HASTA sprint_days HACER
        remaining ← total_story_points - (day * ideal_burn_rate)
        ideal_line.AGREGAR({date: sprint.start_date + day, value: remaining})
    FIN PARA

    // 2. Calcular línea real
    actual_line ← []
    current_date ← sprint.start_date

    MIENTRAS current_date ≤ HOY Y current_date ≤ sprint.end_date HACER
        // Sumar story points de tareas NO completadas al final del día
        points_remaining ← 0
        PARA CADA task EN tasks HACER
            SI task.story.sprint_id = sprint.id ENTONCES
                task_status_at_date ← Obtener_Estado_Historico(task, current_date)
                SI task_status_at_date ≠ "Done" ENTONCES
                    points_remaining ← points_remaining + task.story.story_points
                FIN SI
            FIN SI
        FIN PARA

        actual_line.AGREGAR({date: current_date, value: points_remaining})
        current_date ← current_date + 1 día hábil
    FIN MIENTRAS

    // 3. Calcular proyección (si sprint está activo)
    SI sprint.status = "Active" ENTONCES
        days_elapsed ← actual_line.LENGTH
        days_remaining ← sprint_days - days_elapsed

        SI days_elapsed ≥ 3 ENTONCES  * requiere al menos 3 días de datos *
            // Calcular velocidad real de los últimos 3 días
            points_burned_last_3_days ←
                actual_line[days_elapsed - 3].value - actual_line[days_elapsed - 1].value
            actual_burn_rate ← points_burned_last_3_days / 3

            // Proyectar al final del sprint
            current_remaining ← actual_line[days_elapsed - 1].value
            projected_final ← current_remaining - (actual_burn_rate * days_remaining)

            // Generar alerta si proyección > 0
            SI projected_final > 5 ENTONCES
                Generar_Alerta("Riesgo: Sprint puede no completarse",
                              "Proyección: " + projected_final + " story points sin completar")
            FIN SI
        FIN SI
    FIN SI

    // 4. Construir objeto de retorno
    chart ← {
        sprint_id: sprint.id,
        ideal: ideal_line,
        actual: actual_line,
        projection: projected_final,
        generated_at: TIMESTAMP_ACTUAL
    }

    RETORNAR chart
FIN PROCESO
