🎯 1. Introducción a la Teoría General de Sistemas

📖 Definición

Teoría General de Sistemas (TGS): Marco conceptual desarrollado por Ludwig von Bertalanffy que estudia los sistemas como entidades integradas cuyas propiedades emergentes no pueden entenderse analizando solo sus componentes individuales.

🏗️ Conceptos Fundamentales de la TGS

🌐

🎯

🔄

Procesa entradas para generar salidas

🔗

Los elementos están conectados

📈

Emergencia

Propiedades que surgen del conjunto

⚖️

Homeostasis

Capacidad de autorregulación

💡 Aplicación en Sistemas de Software

En el desarrollo de software, la TGS nos ayuda a:

  • Comprender complejidad: Ver el software como sistema integrado
  • Identificar fronteras: Definir qué incluye y qué excluye el sistema
  • Gestionar interacciones: Entender cómo los módulos se relacionan
  • Optimizar emergencia: Aprovechar propiedades del sistema completo

📊 3. Análisis Estructurado: Fundamentos Teóricos

🎯 Principios del Análisis Estructurado

El análisis estructurado se basa en la descomposición funcional y el enfoque top-down para entender y modelar sistemas complejos.

🔄 Proceso de Análisis Estructurado

✅ Ventajas del Análisis Estructurado

  • Metodología probada: Décadas de uso exitoso
  • Documentación clara: Modelos visuales comprensibles
  • Enfoque gradual: Refinamiento progresivo
  • Separación de aspectos: Lógico vs físico
  • Trazabilidad: Desde requisitos hasta implementación

⚠️ Limitaciones del Análisis Estructurado

  • Orientación a procesos: Dificultad con sistemas orientados a datos
  • Escasa reutilización: Cada análisis es específico
  • Mantenimiento complejo: Cambios requieren múltiples modelos
  • Abstracción limitada: Menos expresivo para sistemas complejos

⚖️ 4. Paradigmas de Análisis: Comparación Teórica

📊 Análisis Estructurado

  • Enfoque: Procesos y funciones
  • Descomposición: Funcional (top-down)
  • Modelos principales: DFD, DER, DD
  • Perspectiva: ¿Qué hace el sistema?
  • Fortaleza: Sistemas de procesamiento
  • Herramientas: Diccionario de datos
  • Evolución: Años 70-80

🎯 Análisis Orientado a Objetos

  • Enfoque: Objetos y responsabilidades
  • Descomposición: Por objetos del dominio
  • Modelos principales: Clases, Casos de Uso, Secuencia
  • Perspectiva: ¿Quién hace qué?
  • Fortaleza: Sistemas interactivos
  • Herramientas: UML, patrones
  • Evolución: Años 90-presente
🔄 Evolución de Paradigmas de Análisis
timeline title Evolución de Paradigmas 1970s : Análisis Estructurado : DFD y DER : Descomposición funcional 1980s : Métodos Estructurados : SSADM : Yourdon/DeMarco 1990s : Orientación a Objetos : UML : Casos de Uso 2000s : Metodologías Ágiles : User Stories : Modeling ágil 2010s+ : Enfoques Híbridos : DDD : Arquitecturas emergentes

🔧 Criterios de Selección de Paradigma

  • Naturaleza del sistema: Procesal vs interactivo
  • Complejidad del dominio: Simple vs complejo
  • Experiencia del equipo: Conocimientos previos
  • Tiempo de desarrollo: Urgencia vs calidad
  • Mantenibilidad: Frecuencia de cambios esperada
  • Reutilización: Necesidad de reutilizar componentes

🛠️ 5. Herramientas del Análisis Tradicional

📊

Diagrama de Flujo de Datos (DFD)

Representa el flujo de información a través del sistema
  • Procesos de transformación
  • Almacenes de datos
  • Entidades externas
  • Flujos de información
  • Niveles de detalle
🗃️

Diagrama Entidad-Relación (DER)

Modela la estructura de datos y sus relaciones
  • Entidades del negocio
  • Atributos de entidades
  • Relaciones entre entidades
  • Cardinalidades
  • Restricciones de integridad
📖

Diccionario de Datos (DD)

Repositorio centralizado de definiciones
  • Definiciones de elementos
  • Estructura de datos
  • Reglas de validación
  • Sinónimos y alias
  • Ubicación y uso
📋

Especificación de Procesos

Describe la lógica de procesos elementales
  • Pseudocódigo
  • Árboles de decisión
  • Tablas de decisión
  • Lenguaje natural estructurado
  • Pre y post condiciones
📅

Diagrama de Transición de Estados

Modela el comportamiento dependiente del tiempo
  • Estados del sistema
  • Eventos disparadores
  • Transiciones válidas
  • Acciones asociadas
  • Estados iniciales/finales
⏱️

Diagrama de Estructura (Hierarchy Chart)

Representa la descomposición jerárquica del sistema
  • Módulos y submódulos
  • Jerarquía de llamadas
  • Parámetros de entrada/salida
  • Interfaces entre módulos
  • Nivel de abstracción

🔧 Integración de Herramientas

En el análisis estructurado, las herramientas se complementan para proporcionar una vista completa del sistema:

  • DFD + DD: Procesos documentados con datos precisos
  • DER + DD: Modelo de datos consistente y detallado
  • DFD + DTE: Comportamiento temporal de procesos
  • Todas las herramientas: Vista 360° del sistema
