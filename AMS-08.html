📚 1. Fundamentos del Control de Versiones

🔍 ¿Qué es el Control de Versiones?

Un sistema que registra los cambios realizados sobre archivos a lo largo del tiempo, permitiendo recuperar versiones específicas, comparar cambios, y colaborar eficientemente en equipos de desarrollo.

🏗️ Evolución de los Sistemas de Control de Versiones
timeline title Historia del Control de Versiones 1970s : Sistemas Locales : SCCS : RCS 1980s : Sistemas Centralizados : CVS : Subversion (SVN) 2000s : Sistemas Distribuidos : BitKeeper : Git (2005) : Mercurial 2010s+ : Plataformas Colaborativas : GitHub : GitLab : Azure DevOps

🖥️ Sistemas Locales

Control de versiones en un solo equipo usando copias de archivos con timestamps.

✅ Simple
❌ No colaborativo, propenso a errores

🏢 Sistemas Centralizados

Servidor central que contiene todos los archivos versionados (CVS, SVN).

✅ Control central, colaboración
❌ Punto único de falla, requiere conexión

🌐 Sistemas Distribuidos

Cada cliente es un repositorio completo con historial completo (Git, Mercurial).

✅ Sin punto único de falla, trabajo offline
❌ Mayor complejidad inicial

🛠️ 2. Git: Arquitectura y Conceptos Fundamentales

📋 Conceptos Clave de Git

  • Repository (Repo): Directorio que contiene el proyecto y su historial
  • Working Directory: Copia local de los archivos del proyecto
  • Staging Area: Área intermedia donde se preparan los cambios
  • Commit: Snapshot del proyecto en un momento específico
  • Branch: Línea de desarrollo independiente
  • Merge: Integración de cambios entre branches

# Inicializar un nuevo repositorio Git
$ git init

🎯 Estados de Archivos en Git

  • Untracked: Archivos nuevos no versionados
  • Modified: Archivos cambiados pero no en staging
  • Staged: Archivos listos para commit
  • Committed: Archivos guardados en el repositorio

🌊 Git Flow

Workflow robusto con ramas main, develop, feature, release y hotfix.

✅ Estructura clara, ideal para releases planificados
❌ Complejo, puede ser lento para equipos ágiles

🚀 GitHub Flow

Workflow simple: main + feature branches con pull requests.

✅ Simple, deploy continuo, perfecto para web
❌ Menos control para productos complejos

🎯 Feature Branch

Cada feature en su propia rama antes de merge a main.

✅ Desarrollo aislado, colaboración limpia
❌ Puede crear conflicts si las ramas son long-lived
# Crear y cambiar a nueva rama feature
$ git checkout -b feature/user-authentication
# Cambiado a nueva rama 'feature/user-authentication'

# Publicar la rama en el repositorio remoto
$ git push -u origin feature/user-authentication
# Rama 'feature/user-authentication' configurada para seguir rama remota

# Cambiar de vuelta a la rama main
$ git checkout main
# Cambiado a rama 'main'

# Integrar los cambios de la feature
$ git merge feature/user-authentication
# Actualizando a1b2c3d..4e5f6g7
# Fast-forward
#  auth.js | 45 +++++++++++++++++++++++++++++++++++
#  1 file changed, 45 insertions(+)

🤝 4. Colaboración y Code Review

📋 Proceso de Code Review

1

Crear Pull Request

Desarrollador crea PR con descripción clara y contexto

2

Asignar Reviewers

Se asignan reviewers apropiados basado en expertise

3

Revisar Código

Reviewers analizan calidad, funcionalidad y estilo

4

Feedback y Discusión

Comentarios constructivos y conversación sobre mejoras

5

Aprobación y Merge

Una vez aprobado, se integra a la rama principal

🎯 Qué Revisar en Code Review

  • Funcionalidad: ¿El código hace lo que debe hacer?
  • Legibilidad: ¿Es fácil de entender y mantener?
  • Diseño: ¿Está bien estructurado y sigue patrones?
  • Tests: ¿Hay suficiente cobertura de pruebas?
  • Performance: ¿Hay posibles cuellos de botella?
  • Seguridad: ¿Existen vulnerabilidades potenciales?

💬 Buenas Prácticas para Code Review

  • Sea constructivo: Enfóquese en el código, no en la persona
  • Explique el "por qué": No solo señale problemas, explique razones
  • Sugiera soluciones: Ofrezca alternativas cuando sea posible
  • Reconozca buen código: Elogie las buenas implementaciones
  • Mantenga conversaciones: Use el PR para discutir y aprender

🛠️ Configuración CI con GitHub Actions

# .github/workflows/ci.yml
name: CI Pipeline
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test
      - run: npm run build

🔧 Beneficios de CI con Git

  • Detección temprana: Errores encontrados rápidamente
  • Automatización: Tests y builds automáticos
  • Feedback rápido: Desarrolladores notificados inmediatamente
  • Calidad consistente: Estándares aplicados automáticamente
  • Deploy seguro: Solo código que pasa tests llega a producción

🚨 7. Troubleshooting y Recuperación

🚨 Problema: Merge Conflict

Dos desarrolladores modificaron las mismas líneas de código.

Solución: 
$ git merge feature-branch
# Auto-merging file.js
# CONFLICT (content): Merge conflict in file.js

$ git status
# both modified: file.js

# Editar file.js manualmente, resolver conflictos
$ git add file.js
$ git commit -m "Resolve merge conflict"

⏪ Problema: Necesito Deshacer el Último Commit

Se hizo commit de código con errores que necesita corrección.

Soluciones: 
# Deshacer commit pero mantener cambios en working directory
$ git reset --soft HEAD~1

# Deshacer commit y cambios (PELIGROSO)
$ git reset --hard HEAD~1

# Crear nuevo commit que deshace cambios
$ git revert HEAD

🔍 Problema: Encontrar Cuándo se Introdujo un Bug

Necesito encontrar en qué commit se introdujo un problema específico.

Solución con Git Bisect: 
$ git bisect start
$ git bisect bad
$ git bisect good v1.0
# Bisecting: 12 revisions left to test after this

# Probar el código, marcar como good o bad
$ git bisect good
# Bisecting: 6 revisions left to test after this

# Continuar hasta encontrar el commit problemático
$ git bisect reset

🔧 Resolución Básica de Conflictos

$ git status
# both modified:   src/auth.js

# Ver conflictos en el archivo
$ git diff
# <<<<<<< HEAD
# código de tu rama
# =======  
# código de la otra rama
# >>>>>>> feature/auth

# Después de resolver manualmente
$ git add src/auth.js
$ git commit -m "Resolve merge conflict in auth.js"

🔐 13. Seguridad en Git

🛡️ Buenas Prácticas de Seguridad

  • No commitear secretos: API keys, passwords, tokens
  • Usar SSH keys: Más seguro que HTTPS con contraseña
  • Firmar commits: Verificar identidad del autor
  • Branch protection: Proteger ramas importantes
# Configurar firma de commits con GPG
$ git config --global commit.gpgsign true

# Verificar commits firmados
$ git log --show-signature

# Buscar secretos accidentales
$ git log --all --grep="password\|secret\|key"

🔍 Herramientas de Escaneo

  • git-secrets: Prevenir commits con secretos
  • TruffleHog: Buscar secretos en historial
  • GitLeaks: Scanner de secretos ligero

💼 14. Mejores Prácticas

✅ Resumen de Mejores Prácticas

  • Commits frecuentes y pequeños: Facilita el tracking y rollback
  • Mensajes descriptivos: Explica el "por qué", no solo el "qué"
  • Branching strategy: Usa un workflow consistente en el equipo
  • Pull requests: Siempre revisar código antes de merge
  • Tests automatizados: CI/CD con hooks y pipelines
  • Documentación: README claro y actualizado
